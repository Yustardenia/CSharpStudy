# 题库新增规则

这套题库允许继续扩容，但新增时要遵守下面的规则，保证页面、筛选器和本地状态都能继续正常工作。

## 改哪里

- 只改 [questions.js](/E:/C#Study/InterviewPrep/questions.js)。
- 往现有分类加题时，直接在对应数组末尾追加对象：
  - `unityLifecycle`
  - `unitySystems`
  - `unityPerformance`
  - `unityProject`
  - `csharpBasic`
  - `csharpAdvanced`
- 不要手写 `id`，题号会自动生成。

## 单题结构

每道新增题都必须保留这 5 个字段，字段名不要改：

```js
{
  difficulty: "基础", // 基础 | 进阶
  tags: ["Awake", "Start", "初始化"],
  prompt: "请你说明 Awake 和 Start 的职责边界。",
  intent: ["考察生命周期顺序", "考察初始化拆分能力"],
  followUps: ["如果对象是运行时实例化的会怎样？", "哪些逻辑放在 Start 更稳？"]
}
```

硬性要求：

- `difficulty` 只能是 `基础` 或 `进阶`
- `tags` 至少 2 个，建议 2-5 个
- `intent` 至少 2 条
- `followUps` 至少 2 条
- `prompt` 必须是一道明确的面试问答题
- 不新增 `answer` 字段

## 写题规则

- 必须用原创中文表述，不直接复制网页或文档原句。
- 一题只问一个核心主题，不把多个独立知识点硬塞成一题。
- `prompt` 要像真实面试提问，不要写成目录标题。
- `intent` 写“这题在考什么”，不要重复题干。
- `followUps` 写“面试官可能继续追什么”，不要偏题。
- 项目题优先问“问题、定位、方案、收益”这类真实追问链路。

## 分类规则

- 优先复用现有分类，不轻易新开分类。
- 当前优先分类：
  - `运行机制与生命周期`
  - `组件通信 / 场景 / Prefab / UI / 物理`
  - `性能 / 资源 / ObjectPool / Addressables / 调试`
  - `项目场景与追问`
  - `基础 / OOP / 集合 / 异常`
  - `进阶 / 泛型 / 委托事件 / async-await / 内存`
- 只有在现有分类明显装不下时，才新建分类。

如果一定要新建分类：

1. 先新增一个数组，例如 `var unityTools = [ ... ];`
2. 再在 `window.INTERVIEW_BANK = []` 这段拼接逻辑后面追加一组：

```js
.concat(buildGroup("U", "Unity", "工具链 / 编辑器扩展", 1, unityTools))
```

说明：

- 第 4 个参数现在只是兼容旧写法保留，传 `1` 即可。
- 真正的题号会按 `U-xxx` / `C-xxx` 自动顺延，不依赖这个数字。

## 题号规则

- Unity 题自动生成 `U-xxx`
- C# 题自动生成 `C-xxx`
- 前面数组多加题后，后面的编号自动顺延，这是正常行为
- 不要在其他文件里写死“某题永远是固定编号”

## 新增后自查

新增题目后，至少检查下面这些项：

1. 打开 [index.html](/E:/C#Study/InterviewPrep/index.html)，页面能正常加载。
2. 新题能在对应分类和关键词搜索里找到。
3. 随机抽题、收藏、错题、草稿保存仍然正常。
4. 顶部 `Unity / CSharp / 总题量` 会自动更新。
5. 控制台没有新增脚本报错。

## 最稳的新增方式

如果只是正常扩题，按这个顺序做：

1. 找到最合适的现有分类数组。
2. 在数组末尾追加题目对象。
3. 不改 `app.js`，不改 `index.html`，不手写 `id`。
4. 打开页面检查新题是否能被筛选和搜索到。

只有在“现有分类已明显不适合承载新主题”时，才新增分类。
