# 12.整数转罗马数字

## 题目类型

- 字符串
- 模拟
- 贪心

## 我的第一反应

- 这题规则很多，第一眼不知道该从哪里下手。
- 一开始容易陷进“4、9、40、90、400、900 要分别怎么判断”的细碎分类里。
- 当时没想到可以先把所有合法数值和罗马数字提前列成表，再按顺序处理。

## 我的难点

- 没有把这题看成“按固定规则表做转换”，而是先把注意力放在各种特殊情况上。
- 对“从大到小不断取当前能用的值”这个过程还不够熟，看到贪心题容易发怵。
- 写代码时边界不够敏感，把 `>=` 写成了 `>`，导致刚好相等时进不去。

## 我这次的错误

- 思路错误：
  没有第一时间想到先列映射表，把题目从“分类讨论”变成“照表转换”。
- 代码错误：
  把 `if (num >= values[i])` 写成了 `if (num > values[i])`。
- 边界错误：
  当 `num` 刚好等于某个数值时，比如 `4`、`9`、`1000`，也应该立刻使用对应符号。

## 正确思路

- 先把所有合法的罗马数字组合按数值从大到小列出来。
- 这些值不只是 `1000、500、100...`，还要把减法形式一起放进去：
  `900 -> CM`、`400 -> CD`、`90 -> XC`、`40 -> XL`、`9 -> IX`、`4 -> IV`。
- 然后从前往后遍历这张表。
- 只要当前 `num >= values[i]`，就说明这个符号现在可以用：
  - 把对应罗马数字拼到结果里
  - 把 `num` 减去这个值
- 一直重复，直到 `num == 0` 为止。

用最朴素的话说，这题不是在“推公式”，而是在“照着固定表不断减”。

## 复杂度

- 时间复杂度：`O(1)`
- 空间复杂度：`O(1)`

说明：因为题目范围固定在 `1 <= num <= 3999`，映射表大小也是固定的，所以可以看成常数复杂度。

## 复盘

- 下次看到“规则固定、可枚举”的转换题，先想能不能把规则整理成表，而不是一上来手写很多分支。
- 如果特殊情况本身也是固定的，就直接把它们放进表里，不要在循环里临时现想。
- 写完后重点检查比较符号，尤其是这种“刚好等于时也要处理”的场景。

## 最优解

- 当前最值得记的版本就是：先列出固定映射表，再从大到小处理。
- 这题虽然常被归到贪心，但对现在的你来说，更好记的方式是：
  “准备好合法数值表，然后不断找当前能减的最大值。”
- 这种写法思路直接、代码稳定、面试和 LeetCode 都够用。

```csharp
using System.Text;

public class Solution
{
    public string IntToRoman(int num)
    {
        int[] values = { 1000, 900, 500, 400, 100, 90, 50, 40, 10, 9, 5, 4, 1 };
        string[] symbols = { "M", "CM", "D", "CD", "C", "XC", "L", "XL", "X", "IX", "V", "IV", "I" };

        StringBuilder ans = new StringBuilder();

        for (int i = 0; i < values.Length; i++)
        {
            while (num >= values[i])
            {
                ans.Append(symbols[i]);
                num -= values[i];
            }
        }

        return ans.ToString();
    }
}
```
