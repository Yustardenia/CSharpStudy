# 13.罗马数字转整数

## 题目类型

- 字符串
- 模拟
- 贪心

## 我的第一反应

- 这题一开始容易想成要把 `IV、IX、XL、XC、CD、CM` 全部单独分类讨论。
- 我当时已经想到“当前字符和右边字符比较”，但一写代码就容易写成双指针，结果把下标和循环条件写乱。
- 最开始没有把这题收紧成一句话：
  当前值小于右边值就减，否则就加。

## 我的难点

- 没有先抓住这题唯一的特殊规则：小数写在大数左边时表示减法。
- 容易把它写成“两个指针一起跑”的题，但其实并不是真正的双指针题，只是遍历当前位置时顺便看一眼右边。
- 最后一个字符没有右边，代码里很容易越界，或者漏掉最后一个字符的处理。

## 我这次的错误

- 思路错误：
  一开始容易陷入枚举各种特殊组合，没有先总结统一规则。
- 代码错误：
  写成了 `i`、`j` 两个指针一起推进，但循环条件没有和字符串长度绑定。
- 边界错误：
  最后一个字符没有右边字符，不能继续访问 `s[j]`。
- 实现错误：
  `for (; i <= j; i++)` 这个条件不对，它控制不了是否遍历完整个字符串。

## 正确思路

- 先准备一张字符到数值的映射表：
  - `I -> 1`
  - `V -> 5`
  - `X -> 10`
  - `L -> 50`
  - `C -> 100`
  - `D -> 500`
  - `M -> 1000`
- 然后从左到右遍历字符串。
- 对于当前位置 `s[i]`：
  - 如果它右边还有字符，并且 `当前值 < 右边值`，说明当前值应该被减掉。
  - 否则就把当前值加到答案里。

用最朴素的话说：

- `VI` 是 `5 + 1`
- `IV` 是 `-1 + 5`
- `MCMXCIV` 是 `1000 - 100 + 1000 - 10 + 100 - 1 + 5`

所以这题不是在背很多特殊情况，而是在统一处理：
比右边小就减，否则就加。

## 复杂度

- 时间复杂度：`O(n)`
- 空间复杂度：`O(1)`

其中 `n` 是字符串长度。

## 复盘

- 下次看到这种“有一点特殊规则的字符串转换题”，先想能不能提炼成统一判断，而不是一上来枚举全部特殊情况。
- 只要题目本质是“遍历当前位置，顺便看一眼右边”，就优先想单下标遍历，而不是双指针。
- 写代码时重点检查：
  - 最后一个字符怎么处理
  - 会不会访问 `i + 1` 越界
  - 循环条件是不是和字符串长度真正绑定了

## 最优解

- 当前最值得记的版本就是：单下标遍历，比较当前值和右边值。
- 如果当前值小于右边值，就减当前值；否则加当前值。
- 这版代码比“手动特判 `IV、IX、XL...`”更短，也更不容易漏情况。

```csharp
public class Solution
{
    public int RomanToInt(string s)
    {
        Dictionary<char, int> dict = new Dictionary<char, int>()
        {
            { 'I', 1 }, { 'V', 5 }, { 'X', 10 }, { 'L', 50 },
            { 'C', 100 }, { 'D', 500 }, { 'M', 1000 }
        };

        int ans = 0;

        for (int i = 0; i < s.Length; i++)
        {
            int current = dict[s[i]];

            if (i < s.Length - 1 && current < dict[s[i + 1]])
            {
                ans -= current;
            }
            else
            {
                ans += current;
            }
        }

        return ans;
    }
}
```

- 这题也可以写成“只遍历到倒数第二个字符，最后一个字符单独加上”的版本。
- 但从“容易理解”和“不容易写错”来看，上面这版更适合作为当前阶段的主记忆版本。
